## BPM Suite란

- BPM Lifecycle 전체 단계를 지원하는 IT 도구 집합
- 프로세스 경영의 개념이 확장되면서 용어도 확장됨

![BPM 용어 확장 단계](img/2025-10-27-20-54-28.png)

## BPM Suite 구성 요소

![구성 요소](img/2025-10-27-20-56-13.png)

1. `BPA`(Business Process Analysis)
    - 비즈니스 프로세스를 발견/식별하여 `모델링`하고, 요구사항을 충족하는지 `분석`하는 솔루션
    - 비즈니스 프로세스를 어떻게 모델링할 것인가? 가 핵심
2. `BPE`(Business Process Execution)
    - 설계된 모델을 `자동화`하여 `실행`하는 솔루션
        - `실행 프로세스 정의`, `자동화 실행`, `모니터링`, `실행 결과 분석`
    - `Camunda Web Modeler`: 프로세스 실행에 필요한 각종 요소의 `디자인`을 위한 도구
    - `Camunda Zeebe Orchestration Engine`: `자동화 실행` 서비스를 제공하는 엔진
    - `Camunda Tasklist`: Human process에서 해야하는 일 리스트
    - `Camunda Operate`: 실행중인, 완료된, 일시중지된 프로세스(`instance`)를 `모니터링` 하는 도구
        - 일시 중지, 취소, 기록 삭제 가능
    - `Camunda Optimize`: 프로세스 진행 이력 기반 `분석 도구`
3. `EAI & ESB`(Enterprise Application Integration, Enterprise Service Bus)
    - 이질적 어플리케이션들을 프로세스 기반으로 `연계/통합`하는 솔루션

![EAI & ESB](img/2025-10-27-21-02-40.png)

4. `BRE`(Business Rule Engine)
    - `어플리케이션 로직`과 `비즈니스 룰`을 `분리`하여 별도로 관리
5. `BAM`(Business Activity Monitoring)
    - `이벤트`를 수집하여 `실시간 경영 의사결정`을 지원하는 솔루션 - `RTE` 실현을 위한 도구

## 프로세스 자동화 구축 및 운영 과정

1. `Design & Deploy`
    - BPE에서 정의하는 실행 가능한 형태
    - Design 요소
        1. `실행가능`한 자동화 프로세스 모델
        2. `IT Application`(ERP 모듈, `Form`)
        3. `프로세스 데이터 모델`(프로세스 변수)
        4. `업무 규칙(Business Rule)`
        5. `업무 담당자` / 역할 지정
            - 실습에서는 생략하라 하심
        6. `통합 인터페이스(service task)`
            - 실습에서는 쓰면 가점
2. `Automation & Integration`
    - `BPM 엔진`이 담당, 사람 / 사람, 사람 / 시스템, 시스템 / 시스템 통합 지휘
    - 자동화 대상 프로세스 유형:
        - 전통적인 `워크플로우 형태 프로세스`
        - `시스템 통합` 중심 프로세스: 사람의 개입이 최소화 되는 것
        - `RPA-Centric` Process
        - `AI-Centric` Process
        - `Event-driven` / Data-driven Process
        - End-to-End Orchestrated Process
3. `Monitor & Control`
    - `BAM` 솔루션 등장(Comunda 내장 기본적인게 있긴 함)
4. `Analysis & Report`
    - `OLAP` 도구 이용
        - `KPI` 형태로 사전 정의된 성과지표를 측정할 수 있음
    - Camunda Optimize 사용 가능
5. `Simulation & Optimize`
    - `시뮬레이션`: 매우 효과적인 `최적화` 방법
        - 가상으로 프로세스를 실행시켜 문제점 도출 및 분석 도구
    - `최적화`: 분석을 통해 발견된 `문제점`을 `해결`
        - 개선기회 모색을 통해 수립된 `개선 방안`을 `적용`하여 해당 프로세스가 최적화 상태로 실행될 수 있도록 함

## AI - RPA - BPM 통합

- `Hyperautomation`: 모든 자동화 기술을 하나의 오케스트레이션 플랫폼에서 통합 관리하는 차세대 자동화 패러다임
- 구성 요소:
    - `BPM 엔진`
    - `RPA 봇`
    - `AI Agent`
    - `API / Connector / Event`
    - `Process Minig / Optimize`
- 목표: 
    - `지능형 / 자가개선형` 프로세스 발전
    - 조직의 `End-to-End 디지털 전환` 가속화

## 시스템 통합 문제

1. 메인 프레임 시대 -> 유닉스 시스템
    - `p2p`로는 한계가 있다.
2. `EAI` 등장: `엔터프라이즈 미들웨어`로 통합
3. `ESB`로 진화

- To-Be 아이디어: 내부 인원 파악 자동화
- 현재 문제점:
    1. 내부 직원과 입장 안내 직원의 의사소통 과정에서 오류가 발생할 가능성이 있음
    2. 현장 인원 수가 실제 파악한 인원이랑 다를 수 있음
    3. 내부 직원에게 전달된 인원 수가 실제 인원과 다를 경우 내부가 혼란스러워질 수 있음
    4. 실제로 김형훈이 올렸던 html 파일에는 비슷한 사례가 있음.
- 해결 방안:
    1. 입장 시 QR 코드를 스캔하여 DB에 자동으로 인원 수 기록
        - 입장 가능한 인원 수가 나오면 자동으로 QR 코드가 생성되서(혹은 프린트 되어) 고객이 알아서 찍고 들어간다.
    2. 퇴장 시 (출구랑 입구가 분리된 공간 가정)
        1. 퇴장 시에도 QR 코드를 스캔하여 DB에서 인원 수 차감
            - 제일 간단한 방법이지만, 나갈 때도 QR 코드를 찍는 거를 대체로 번거로워 함
        2. 입장 가능 시간을 5분으로 정해놓고 내부에서 활동을 하면 다시 5분 갱신, 5분 동안 아무것도 안하면 퇴장 처리(access token 개념)
            - 당연히 이 방법은 적절하지 않다고 본다.
        3. 퇴장 시에는 출구의 스캐너가 자동으로 인식하는 방법
            - 구현하는 방법이 간단한지 잘 모르겠지만, 이것보다는 더 좋은 방법이 있을 것 같다.
            - 입장 등록 인원이랑 실제 입장 인원이 다를 수 있음 (ex. 영유아는 입장 등록 인원으로 포함 안할 수 있다.). 이때 스캐너로 인식하면 더 많은 인원을 퇴장처리 할 수도 있다.
        4. 출구에 직원 배치
            - 이건 To-Be라고 할 수 없을 듯
        - 이 방안 중에서는 1번이 제일 현실성이 있어 보인다. 다만, 이 방법을 하려면 고객이 번거롭다고 느끼지 않게 해야 한다.
